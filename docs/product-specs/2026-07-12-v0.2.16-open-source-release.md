# FrameQ v0.2.16 Open-Source Release Spec

## Status

Approved for implementation on 2026-07-12.

## Release Intent

FrameQ v0.2.16 is a stable GitHub Release for a personal-development, small-user, and
open-source distribution model. It is not a notarized commercial macOS distribution. The release
must preserve the existing local-first, lightweight-installer, SourceIdentity, entitlement, and
disabled-WeChat-payment boundaries.

## Supported Artifacts

- Windows x64 NSIS installer plus the existing Tauri updater artifacts and `latest.json`.
- macOS Intel (`x86_64`) DMG.
- macOS Apple Silicon (`aarch64`) DMG.
- No Linux artifact, Universal macOS binary, ASR model weight, LLM credential, payment credential,
  user configuration, or user data is included.

## macOS Signing and User Trust Model

- Both macOS app bundles use the repository's existing ad-hoc signing path and must pass
  `codesign --verify --deep --strict` before DMG upload.
- Ad-hoc signing verifies bundle integrity only. It does not establish a Developer ID identity and
  does not satisfy Apple notarization.
- The release may be published as a stable GitHub Release under the approved personal/open-source
  distribution model only when the release notes prominently disclose the one-time Gatekeeper
  approval requirement.
- The primary user instruction is to launch once, dismiss the warning, then use **System Settings
  → Privacy & Security → Open Anyway** on current macOS. Control-click **Open** may be documented
  for older macOS. `xattr -dr com.apple.quarantine /Applications/FrameQ.app` is a clearly labelled
  terminal fallback, not the default instruction.
- FrameQ must not claim that the DMGs are Developer ID signed, notarized, App Store reviewed, or
  suitable for silent enterprise deployment.
- Developer ID signing and notarization remain the removal condition for the Gatekeeper warning
  before a broader consumer or commercial distribution.

## Release Flow

1. Update every desktop release version source and lockfile to `0.2.16` with a drift test.
2. Update release documentation and prepare version-specific release notes.
3. Run the complete local project gates on the release commit.
4. Push the verified release commit to `main`, create annotated tag `v0.2.16`, and push the tag.
5. The tag-triggered Desktop Release workflow creates a Draft Release and builds all supported
   artifacts. A draft must not replace `v0.2.15` as the updater-visible latest stable release.
6. Inspect the Draft Release only after the Windows, macOS Intel, and macOS Apple Silicon jobs are
   green. Confirm expected assets, updater metadata, architecture-specific DMGs, bundled runtime
   imports, Deno smoke, and macOS codesign verification.
7. Replace the generic generated release body with the reviewed v0.2.16 notes, including the
   macOS Gatekeeper disclosure, manual macOS update behavior, and disabled WeChat Pay status.
8. Publish the Draft as a non-prerelease stable release only after every gate above succeeds.

## Failure and Rollback Rules

- A failed or incomplete Draft Release remains non-public and must not update `latest.json` for
  stable clients.
- Failed jobs may be repaired and rerun against the same release commit only when artifacts remain
  version-consistent. Do not silently mix binaries built from different commits.
- If a release tag or asset set is incorrect, keep `v0.2.15` published as latest while the draft is
  corrected or removed. Never force-move a tag that users may already have fetched.
- Publishing the Draft is a separate irreversible external action and occurs only after artifact
  inspection.

## Acceptance Criteria

- `app/package.json`, `app/package-lock.json`, `app/src-tauri/Cargo.toml`,
  `app/src-tauri/Cargo.lock`, and `app/src-tauri/tauri.conf.json` agree on `0.2.16` wherever they
  carry the application version.
- App tests/build, Rust tests, worker tests/Ruff, server tests/build, script tests, documentation
  validation, and `git diff --check` pass on the release commit.
- The Desktop Release run for `v0.2.16` completes successfully for Windows x64, macOS Intel, and
  macOS Apple Silicon.
- The Draft contains the Windows installer/updater metadata and two architecture-labelled macOS
  DMGs built from the tagged commit.
- Release notes explain that macOS packages are ad-hoc signed and not notarized before users reach
  the download links.
- `WECHAT_PAY_ENABLED` remains closed by default; the release does not request real payment or LLM
  credentials and does not describe WeChat Pay as available.

## Residual Risk

Users downloading either macOS DMG will encounter a Gatekeeper warning until Apple Developer ID
signing and notarization are adopted. Hosted Apple Silicon build/runtime verification is required,
but it does not replace a clean-machine manual Apple Silicon acceptance run; that limitation must
remain visible in the release report.
